UML-Klassendiagramme: Die Blaupause deiner Software

Stell dir vor, du baust ein Haus. Du fängst nicht einfach an zu mauern, sondern schaust zuerst auf einen Bauplan. In der Softwareentwicklung ist ein UML-Klassendiagramm genau das: der Bauplan für dein Programm. Es zeigt dir alle Bauteile (Klassen) und wie sie miteinander verbunden sind. Für die IHK-Prüfung ist es super wichtig, dass du diese Pläne lesen und verstehen kannst.

Eine Klasse ist wie ein Rezept oder ein Bauplan. Sie beschreibt, welche Eigenschaften (Attribute) und Fähigkeiten (Methoden) ein Objekt später haben wird. Im Diagramm hat eine Klasse drei Bereiche:

  • Oben: Der Klassenname (z.B. `Akku`).
  • Mitte: Die Attribute (z.B. `-id: int`). Das Minus-Zeichen - bedeutet private (nur die Klasse selbst darf zugreifen). Ein Plus + bedeutet public (jeder darf zugreifen).
  • Unten: Die Methoden (z.B. `+addMessreihe(...)`).

Beispiel aus IHK-Prüfung W24/25: Die Klasse `Akku`.

# So wird der Plan in Ruby-Code umgesetzt:
class Akku
  # -id: int -> wird zu @id (private)
  # Mit attr_reader machen wir eine öffentliche Lese-Methode (Getter).
  attr_reader :id
  
  # -nennkapazitaet: int -> wird zu @nennkapazitaet
  attr_accessor :nennkapazitaet

  def initialize(id, nennkapazitaet)
    @id = id
    @nennkapazitaet = nennkapazitaet
    @messreihenliste = [] # Speichert die Beziehung zu Messreihe
  end

  # +addMessreihe(...)
  def add_messreihe(messreihe)
    @messreihenliste << messreihe
  end
end

Das Wichtigste an UML sind die Verbindungen zwischen den Klassen. Es gibt zwei Hauptfamilien von Beziehungen:

Typ 1: Vererbung ("Ist-ein")

Symbol: Pfeil mit leerer, geschlossener Spitze (──▷)

Das ist eine "Ist-ein"-Beziehung. Die Unterklasse ist eine spezielle Version der Oberklasse und erbt ihre Eigenschaften.
Alltagsbeispiel: Ein `PKW` ist ein `Fahrzeug`.

IHK-Beispiel (W24/25): Ein Strom-Messwert ist ein Messwert.

# "class Strom < Messwert" bedeutet: Strom erbt von Messwert.
class Strom < Messwert
  # ... spezielle Dinge nur für Strom ...
end
Typ 2: "Hat-ein" / "Teil-von" Beziehungen

Hier gibt es drei Abstufungen, von schwach nach stark. Sie zeigen, wie eng Klassen miteinander verbunden sind.

a) Assoziation (Lose Verbindung)
Symbol: Einfache Linie (───)

Eine "Kennt-ein"-Beziehung. Die Klassen sind Partner und existieren unabhängig voneinander.
Alltagsbeispiel: Ein `Lehrer` kennt ein `Fach`. Wenn der Lehrer die Schule verlässt, existiert das Fach weiter.

b) Aggregation (Teile können ohne das Ganze existieren)
Symbol: Linie mit einer leeren Raute (───◇)

Eine "Hat-ein"-Beziehung. Das Ganze besteht aus Teilen, aber die Teile können auch ohne das Ganze existieren.
Alltagsbeispiel: Ein `Team` hat `Spieler`. Löst sich das Team auf, sind die Spieler immer noch da und können in anderen Teams spielen.

c) Komposition (Teile sind vom Ganzen abhängig)
Symbol: Linie mit einer gefüllten Raute (───◆)

Eine starke "Teil-von"-Beziehung. Die Teile können ohne das Ganze nicht existieren. Wird das Ganze gelöscht, sind die Teile auch weg.
IHK-Beispiel (W22/23): Ein Angebot besteht aus Bestellpositionen. Löschst du das Angebot, sind die Positionen darin sinnlos und werden mitgelöscht.

Übersichtstabelle zum Vergleichen
Beziehungstyp Symbol Beschreibung Teil existiert alleine?
Assoziation ─── Kennt-ein (Partner) Ja
Aggregation ───◇ Hat-ein (lose Teile) Ja
Komposition ───◆ Besteht-aus (abhängige Teile) Nein

Ein Interface ist ein Vertrag. Es schreibt vor, welche Methoden eine Klasse haben muss, aber nicht, wie diese Methoden funktionieren. Klassen können ein Interface "implementieren", um zu versprechen, dass sie sich an diesen Vertrag halten.

Symbol: Ein Kasten mit dem Zusatz <<interface>> und einer gestrichelten Pfeillinie (- - -▷) zur implementierenden Klasse.

IHK-Beispiel (S22): Das Interface ImportService definiert, dass jeder Importer (z.B. für verschiedene Hersteller-Dateien) die gleichen Methoden wie Read() oder GetDriveName() haben muss. So kann das Hauptprogramm jeden Importer auf die gleiche Weise benutzen, egal wie er intern funktioniert.

# In Ruby simulieren wir Interfaces oft mit Modulen.
# Das ist der "Vertrag":
module ImportService
  def read
    raise NotImplementedError, "Jeder Importer muss eine 'read' Methode haben!"
  end

  def get_drive_name
    raise NotImplementedError, "Jeder Importer muss 'get_drive_name' implementieren!"
  end
end

# Diese Klasse "unterschreibt" den Vertrag:
class Manufacturer1Importer
  include ImportService # Verspricht, die Regeln von ImportService zu befolgen

  def read
    # ... hier steht der Code, um Datei von Hersteller 1 zu lesen ...
  end

  def get_drive_name
    # ... hier steht der Code, um den Namen auszulesen ...
  end
end

Wenn ein Klassenname oder eine Methode kursiv geschrieben ist, bedeutet das abstrakt. Das ist ein Befehl an den Programmierer!

  • Abstrakte Klasse: Du darfst von dieser Klasse kein Objekt erzeugen. Sie ist nur eine Schablone für ihre Unterklassen. (IHK-Frage W24/25, Aufgabe 1.1)
  • Abstrakte Methode: Diese Methode hat keinen Code in der Oberklasse. Jede Unterklasse ist gezwungen, eine eigene Implementierung für diese Methode zu schreiben.

IHK-Beispiel (W24/25): Die Klasse Messwert ist abstrakt. Ihre Methode pruefeWert() ist ebenfalls abstrakt. Das bedeutet: Jeder konkrete Messwert (wie Strom) MUSS eine eigene `pruefe_wert`-Methode haben, die genau sagt, wann ein Stromwert gültig ist.

Übungsaufgabe im IHK-Stil

Aufgabe (basierend auf IHK-Prüfung W24/25, Aufgabe 1.4): Schau dir das Diagramm an. Die Aufgabe lautet:

"Jeder Messplatz kennt mehrere Akkus. Um den Alterungsprozess eines Akkus dokumentieren zu können, werden für jeden Akku mehrere Messreihen aufgenommen. Jede dieser Messreihen wiederum enthält mehrere Messwerte."

Deine Aufgabe: Erkläre die Beziehungen zwischen den Klassen und nenne die Multiplizitäten und die "Rollennamen" (also die Namen der Variablen, die die Beziehung speichern).

Lösungs-Check:

  • Messplatz hat 1..* Akkus. Die Beziehung wird in der Variable akkuliste gespeichert. (Assoziation)
  • Akku hat 0..* Messreihen. Die Beziehung wird in der Variable messreihenliste gespeichert. (Assoziation)
  • Messreihe besteht aus 1..* Messwerten. Die Beziehung wird in der Variable messwertliste gespeichert. (Hier könnte man über Komposition argumentieren, da Messwerte ohne ihre Reihe kaum Sinn ergeben).
